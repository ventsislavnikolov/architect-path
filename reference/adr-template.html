ADR Template

The skeleton of every Architecture Decision Record. Keep it one page. Capture the why.

Source the structure from Michael Nygard's original and the ADR community. The shape below is the durable minimum.

ADR-NNNN: <short decision in a noun phrase>

Number sequentially. Title = the decision, not the problem. e.g. "ADR-0007: React as standard frontend library".

Status

ProposedAcceptedSuperseded by ADR-NNNNDeprecated

ADRs are immutable. You don't edit a decision — you supersede it with a new ADR and link them.

Context

The forces in play — business and technical — that make this a decision worth recording. What's true now? What pressure forces a choice? No solution yet.

Decision

Active voice: "We will …". One clear choice. This is the load-bearing line.

Options considered

The serious alternatives, each with its trade-off. Proves you saw the landscape — and that the decision was a judgement, not a default. (Optional but high-value for a promotion portfolio.)

Consequences

What becomes true after this decision — positive, negative, and neutral. The negative ones are the sacrifices you named in your trade-off sentence. This section is where architects prove honesty. Never leave it one-sided.

Enforcement / fitness function (extension)

How will this decision be assured over time? An ADR documents; a fitness function enforces. Name the automated check (or write "manual review" honestly).

Rules of thumb One decision per ADR · one page · immutable (supersede, don't edit) · store in-repo next to the code (/docs/adr/) · link to C4 diagrams for the "what".
