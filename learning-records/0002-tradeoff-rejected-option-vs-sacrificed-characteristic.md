# 0002 — Misconception: rejected option ≠ sacrificed characteristic

**Date:** 2026-06-27
**Status:** Active (watch in future lessons)

## Context
First skill rep from Lesson 1. Learner wrote a trade-off statement for a real decision
(choosing React over existing Ember/Angular).

## What the learner produced
"Избрахме React, жертвайки EmberJS или Angular ... защото има голямо community, лесно намиране
на нови хора и може да се ползва за нативен мобилен app."

## Key insight / misconception observed
- **Confused the rejected alternative with the sacrificed characteristic.** Put the competitor
  (Ember/Angular) in the "trading away" slot instead of a quality given up. → The whole point of
  the lesson. Grammar must be *trading [characteristic] for [characteristic]*, never *trading [competitor]*.
- **Stated only upsides — no downside named.** The "senior dev / best pick" failure mode the lesson warns about.

## What the learner did WELL
- The "because" was genuinely business-framed: hiring/talent pool + React Native skill-and-code
  reuse (mobile strategy cost) + community-as-longevity-risk. This half is strong — build on it.

## Coaching given
- Sacrifice angle 1: React = library not framework → gives up built-in structure/conventions.
- Sacrifice angle 2 (stronger, contextual): they already run Ember/Angular → React sacrifices
  org-wide frontend-stack consistency (now maintaining multiple stacks). Panel pounces on this.
- Provided a model rewrite with explicit sacrifice + "outweighs the cost" verdict.

## Implications for teaching
- Lesson 2 (ADR) should reuse THIS React decision as the running example — consequences section
  is the natural home for the downsides he omitted.
- Reinforce "name the cost first" repeatedly; check it returns in later reps (spacing).
