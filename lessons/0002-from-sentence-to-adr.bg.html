От изречение към ADR

Превръщане на trade-off изречението в траен, „повишаващ“ запис — и проверка, която го пази жив.

Защо точно сега. Trade-off изречението ти (Урок 1) е мисъл. Един ADR е артефактът — нещото, към което сочиш панела за повишение: „ето решение, което притежавах, с назована цена и честни последствия.“ Папка с добри ADR-и е видимото влияние, което титлата Архитект награждава.

В Урок 1 написа: „Избрахме React, жертвайки еднаквостта на frontend стека за ecosystem и мобилно преизползване, защото hiring + React Native + community.“ Едно изречение. Сега му даваме дом с още три хода — и секцията, която дели честните архитекти от мажоретките: Consequences.

ADR улавя context-а на едно решение, самото решение и неговите consequences — защо-то, написано веднъж, четено с години.1 Trade-off изречението е семето; секцията Consequences е мястото, където недостатъците най-сетне се назовават изцяло.

Същото решение, като пълен ADR

Построен по ADR template. Забележи — негативите не са скрити, те са целта.

Title

ADR-0007: React като стандартна frontend library

Status

Accepted

Context

Имаме няколко frontend-а на Ember и Angular. Наемането и за двата е бавно, а talent pool-ът се свива. Мобилен app е в roadmap-а. Трябва ни frontend default, който сваля цената на наемане, намалява риска от остаряване и дава достоверен мобилен път — без да пренаписваме всичко наведнъж.

Decision

Ще приемем React като стандартна library за всяка нова frontend работа и като migration цел за съществуващите Ember/Angular app-и, когато следващо ги пипаме съществено.

Options considered

  • Оставаме на Angular/Ember — нула migration, но влошаващ се hiring и без общ мобилен път.
  • React — най-голям talent pool + React Native преизползване; но е library, не framework.
  • Vue/Svelte — силен DX, по-малък hiring pool, по-слаба native-mobile история.

Consequences

  • По-голям talent pool → по-бързо, по-евтино наемане.
  • React Native дава web→mobile преизползване на умения и код → по-ниска цена на мобилна стратегия.
  • Голям ecosystem сваля build-vs-buy цената и риска от остаряване.
  • Отсега въртим три frontend стека, докато legacy се мигрира → фрагментирана експертиза, по-голяма maintenance площ.
  • React е library: routing, state, data-fetching са наши решения → риск от разнобой между екипите.
  • Mitigation: пускаме paved-road React starter (одобрени router, state, lint), за да е безопасният избор и default-ният.

Това носи повишение. Панел, който го чете, вижда човек, който е притежавал решение, прегледал е алтернативи и е бил честен за цената — после я е смекчил. Тази последна част е на архитектурно ниво.

Да залепне: fitness function

ADR документира решението. Шест месеца по-късно някой вдига нов Angular app и решението тихо изгнива. Една fitness function го налага — автоматична проверка в CI.2 За ADR-0007:

# CI: блокирай НОВИ frontend пакети на legacy стекове
# „Без нови Angular/Ember app-и“ — решението, като код.
for pkg in $(git diff --name-only origin/main | grep 'package.json'); do
  if grep -qE '"@angular/core"|"ember-source"' "$pkg" \
     && is_new_package "$pkg"; then
    echo "✗ ADR-0007: новите frontend-и трябва да са React. Виж docs/adr/0007.md"
    exit 1
  fi
done

Сега решението не може да дрейфне тихо — build-ът говори за него. Това е governance, без да стоиш зад гърба на никого: същината на influence без authority.

Направи това сега — твоята осезаема победа

Вземи React решението (или твоето от Урок 1) и напиши пълен ADR по template-а. Минимална летва:

  • ≥ 2 options разгледани, всеки със своя trade-off.
  • ≥ 2 негативни consequences, назовани честно (това се оценява най-строго).
  • Една идея за fitness function — дори едно изречение: „ще го налагаме чрез…“.

Изпрати го обратно. Ще го ревюирам като панел — честна ли е цената, реални ли са алтернативите, наложимо ли е решението? Този ADR е вторият ти артефакт за портфолиото.

Аз съм твоят учител — питай ме каквото и да е. Искаш реална fitness function за твоя CI стек? Не си сигурен колко гранулирано трябва да е решение, за да заслужи ADR? Питай.
