From Sentence to ADR

Growing your trade-off sentence into a durable, promotable record — and a check that keeps it true.

Why this, now. Your trade-off sentence (Lesson 1) is a thought. An ADR is the artifact — the thing you point a promotion panel to: "here is a decision I owned, with the cost named and the consequences honest." A folder of good ADRs is the visible influence the Architect title rewards.

In Lesson 1 you wrote: "We chose React, trading frontend-stack consistency for ecosystem & mobile reuse, because hiring + React Native + community." One sentence. Now we give it a home with three more moves — and the section that separates honest architects from cheerleaders: Consequences.

An ADR captures one decision's context, the decision, and its consequences — the why, written once, read for years.1 The trade-off sentence is the seed; the Consequences section is where its downsides finally get named in full.

The same decision, as a full ADR

Built from the ADR template. Notice the negatives are not hidden — they're the point.

Title

ADR-0007: React as the standard frontend library

Status

Accepted

Context

We run several frontends on Ember and Angular. Hiring for both is slow and the talent pool is shrinking. A mobile app is on the roadmap. We need a frontend default that lowers hiring cost, de-risks longevity, and gives us a credible mobile path — without rewriting everything at once.

Decision

We will adopt React as the standard library for all new frontend work, and as the migration target for existing Ember/Angular apps when they are next touched substantially.

Options considered

  • Stay on Angular/Ember — zero migration, but worsening hiring & no shared mobile path.
  • React — largest talent pool + React Native reuse; but it's a library, not a framework.
  • Vue/Svelte — strong DX, smaller hiring pool, weaker native-mobile story.

Consequences

  • Larger talent pool → faster, cheaper hiring.
  • React Native enables web→mobile skill & code reuse → lower mobile-strategy cost.
  • Large ecosystem lowers build-vs-buy cost and longevity risk.
  • We now run three frontend stacks until legacy is migrated → fragmented expertise, larger maintenance surface.
  • React is a library: routing, state, data-fetching are our decisions → risk of per-team inconsistency.
  • Mitigation: ship a paved-road React starter (approved router, state, lint) so the safe choice is the default choice.

This is promotable. A panel reading it sees a person who owned a decision, surveyed alternatives, and was honest about the cost — then mitigated it. That last part is architect-grade.

Make it stick: the fitness function

An ADR documents the decision. Six months later, someone spins up a new Angular app and the decision quietly rots. A fitness function enforces it — an automated check in CI.2 For ADR-0007:

# CI: block NEW frontend packages on legacy stacks
# "No new Angular/Ember apps" — the decision, as code.
for pkg in $(git diff --name-only origin/main | grep 'package.json'); do
  if grep -qE '"@angular/core"|"ember-source"' "$pkg" \
     && is_new_package "$pkg"; then
    echo "✗ ADR-0007: new frontends must use React. See docs/adr/0007.md"
    exit 1
  fi
done

Now the decision can't drift silently — the build speaks for it. That is governance without you standing over anyone's shoulder: the essence of influence without authority.

  • ≥ 2 negative consequences named honestly (this is graded hardest).
